# Domain model

This document captures the first business model for AI Ticket & Project Follow-up Agent.

It is intentionally lightweight. The goal is to understand the business before generating C# classes, database tables, or API endpoints.

## Core domain concepts

### Ticket

A ticket represents a business follow-up item, task, issue, project action, quotation follow-up, invoice follow-up, visit follow-up, or client/admin request.

Initial ticket fields:

- `Id`
- `Title`
- `Description`
- `Reference`
- `Status`
- `Priority`
- `CreatedByUserId`
- `AssignedToUserId`
- `CreatedAt`
- `UpdatedAt`
- `DueDate`
- `CompletedAt`

Possible future fields:

- `Location`
- `CustomerName`
- `ProjectName`
- `RelatedInvoiceNumber`
- `RelatedQuotationNumber`

These future fields are not part of the first implementation unless the MVP needs them.

### User

A user represents a person who can create, own, update, approve, or review tickets.

The first version should include a simple domain-level user concept, but it should not introduce full authentication yet.

This avoids two problems:

- delaying users completely would make assignment and audit records feel fake;
- adding full ASP.NET Identity too early would pull the project away from the ticket workflow before the core domain is stable.

Initial user fields:

- `Id`
- `DisplayName`
- `Email`
- `Role`
- `IsActive`

Initial roles:

- `Admin`
- `Manager`
- `Agent`
- `Viewer`

ASP.NET Identity integration can come later when authentication and authorization become the session focus. The domain should not directly depend on ASP.NET Identity classes.

Initial user-related questions:

- Who created the ticket?
- Who is assigned to follow up?
- Who approved or rejected an AI suggestion?
- Who performed an audited action?

### Ticket status

Initial statuses:

- `New`
- `InProgress`
- `Waiting`
- `Blocked`
- `Done`

Important rule: status is business meaning, not just a label.

For example:

- `New`: ticket has been created but active work has not started.
- `InProgress`: someone is actively working on it.
- `Waiting`: progress depends on another person or external input.
- `Blocked`: progress cannot continue until a blocker is resolved.
- `Done`: work is complete.

### Ticket priority

Initial priorities:

- `Low`
- `Medium`
- `High`
- `Urgent`

Priority helps users decide what needs attention first. AI may suggest priority, but a human must approve the final business change.

### AI suggestion

An AI suggestion is advice generated by an AI provider or agent workflow.

Examples:

- ticket summary;
- priority recommendation;
- blocker detection;
- next-action suggestion;
- follow-up email/message draft.

An AI suggestion must not automatically change the ticket.

Initial AI suggestion fields:

- `Id`
- `TicketId`
- `SuggestionType`
- `SuggestedValue`
- `Reason`
- `Status`
- `CreatedAt`
- `ReviewedAt`
- `ReviewedByUserId`

Initial suggestion statuses:

- `PendingReview`
- `Approved`
- `Rejected`
- `EditedBeforeApproval`

### Approved business change

An approved business change is a real change to ticket data after human review.

Example:

```text
AI suggestion: "This ticket looks urgent."
Human approval: user approves changing priority from Medium to Urgent.
Business change: ticket priority becomes Urgent.
Audit log: records the AI suggestion, approval, and final priority change.
```

This distinction protects the system from silent AI-driven data changes.

### Audit log

An audit log records important business events and AI-related decisions.

Events to record:

- ticket created;
- ticket title changed;
- ticket description changed;
- ticket assigned or reassigned;
- ticket status changed;
- ticket priority changed;
- ticket marked as blocked;
- ticket blocker removed;
- ticket completed;
- ticket reopened;
- AI summary generated;
- AI priority suggestion generated;
- AI blocker suggestion generated;
- AI next-action suggestion generated;
- AI follow-up message generated;
- AI suggestion approved;
- AI suggestion rejected;
- AI suggestion edited before approval;
- human-approved business change applied.

Initial audit log fields:

- `Id`
- `TicketId`
- `EventType`
- `OldValue`
- `NewValue`
- `PerformedByUserId`
- `PerformedAt`
- `Source`

Initial audit sources:

- `Human`
- `AiSuggestion`
- `ApprovedAiSuggestion`
- `System`

## First business rules

### Ticket creation

- A ticket must have a non-empty title.
- A ticket should explain what the follow-up is about.
- A ticket must have a valid creator.
- A ticket starts with a valid status.
- A ticket starts with a valid priority.
- A ticket may have a due date.
- If a due date is provided, it should not be earlier than the creation date.
- Ticket creation should be audited.

### Ticket assignment

- A ticket may be unassigned at first, depending on the workflow.
- If assigned, the assigned user must be valid.
- Assignment or reassignment should be audited.

### Ticket status changes

- Status changes must use allowed statuses.
- Completing a ticket should set `CompletedAt`.
- Reopening a ticket should clear or update completion state.
- Some fields may become read-only when a ticket is `Done`.
- Status changes should be audited.

### Ticket priority changes

- Priority changes must use allowed priorities.
- AI can suggest a priority.
- A human must approve before AI-suggested priority changes affect the ticket.
- Priority changes should be audited.

### AI suggestions

- AI suggestions are advisory.
- AI suggestions should be stored for review.
- AI suggestions should include a reason when possible.
- AI suggestions must not silently change business data.
- Approval, rejection, or editing of AI suggestions should be audited.

## Open questions

- Should the first version include users as full database entities, or use simple user IDs until authentication is introduced?
- Should `Reference` be a simple text field, or should it later become typed references such as invoice, quotation, visit, or project?
- Should location be part of the first MVP, or only a later extension?
- Which ticket fields become read-only after completion?
- Should blocked tickets have a separate blocker reason field from the start?
- Should `DueDate` become required for all tickets, or optional depending on ticket type?

## Session 2 design decision

Start with the smallest domain that supports the portfolio story:

- Ticket;
- simple domain User;
- Status;
- Priority;
- optional DueDate;
- AI suggestion;
- audit log;
- user references for assignment, approval, and auditing.

Delay full ASP.NET Identity authentication, detailed permissions, and complex reference types until the project has working ticket and AI-suggestion flows.
